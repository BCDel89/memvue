# memvue V2 — Plan

Status: **draft** · 2026-04-20

V1 ships a polished browser: tabs for All Memories, Local Files, and Graph, with semantic search, pluggable adapters, mobile-responsive UI, and `mem0://` cross-links. The consolidation experiment (704 → 52 memories) exposed the next layer of problems: **mem0's dedup is opaque, metadata is fragile, and nothing surfaces drift or staleness**.

V2 is about *curating* what's there, *ingesting* new content cleanly, and *opening the door* to a wider audience with optional AI features and light monetization hooks. Self-hosted core stays MIT forever.

---

## Goals

1. Hygiene tools: consolidation UI, duplicate detection, staleness flags, metadata editor.
2. Pluggable LLM layer: optional AI features via any provider (Ollama, OpenAI-compatible, Anthropic).
3. Ingest pipeline: drop files / URLs / text → extracted structured memories.
4. Better editing: proper markdown editor, chunk reassembly, metadata-preserving updates.
5. Light analytics: counts, growth, coverage gaps.
6. Foundations for a hosted future: multi-tenancy, feature flags, export/backup.
7. Monetization hooks: Sponsor link, sponsorware gate, affiliate-ready Settings.

---

## Non-goals

- Full-fledged wiki / Notion clone.
- Multi-user auth in V2 (design for it, don't build it yet).
- Hosted memvue.app (Phase 3, separate project).
- Memory → Linear integration (one-off; add as a plugin later).
- Memory diff visualizer (niche, low ROI).

## Architecture additions

### LLMAdapter

Mirrors `MemoryAdapter`. Lives in `backend/llm/`.

```python
from abc import ABC, abstractmethod

class LLMAdapter(ABC):
    name: str

    @abstractmethod
    async def extract(self, text: str, schema: dict) -> list[dict]: ...

    @abstractmethod
    async def tag(self, content: str, taxonomy: list[str] | None = None) -> list[str]: ...

    @abstractmethod
    async def summarize(self, texts: list[str], style: str = "bullet") -> str: ...

    @abstractmethod
    async def health(self) -> bool: ...

    def capabilities(self) -> set[str]:
        return {"extract", "tag", "summarize"}
```

Providers:

- `OllamaAdapter` — base URL, model name; uses `/api/chat`.
- `OpenAICompatibleAdapter` — covers OpenAI, OpenRouter, Groq, Together, LM Studio. Base URL + key + model.
- `AnthropicAdapter` — direct API.

Registered in `backend/main.py`. Configured via env vars *or* Settings UI (`runtime-config.json`). If no adapter is configured, `capabilities()` returns `set()` and AI-dependent UI renders an empty-state with "Configure an LLM in Settings."

### Feature flags

New module `backend/features.py`:

```python
FEATURES = {
    "ai_ingest":     {"requires": "llm", "sponsor_only": False},
    "ai_tagging":    {"requires": "llm", "sponsor_only": True},
    "ai_digest":     {"requires": "llm", "sponsor_only": True},
    "consolidation": {"requires": None,  "sponsor_only": False},
    "duplicates":    {"requires": None,  "sponsor_only": False},
    "staleness":     {"requires": None,  "sponsor_only": False},
    "analytics":     {"requires": None,  "sponsor_only": False},
}
```

`GET /features` returns the active feature map based on configured adapters + sponsor state. Frontend queries this at mount and hides disabled features cleanly. `sponsor_only` features are fully functional in self-hosted MIT builds — the flag only applies if a future hosted version ever uses it.

### Metadata-preserving update path

Fix the V1 landmine: `mem0.Memory.update()` strips metadata. Add a backend helper:

```python
async def update_content_preserving_meta(adapter, memory_id, new_content):
    existing = await adapter.get(memory_id)
    await adapter.delete(memory_id)
    return await adapter.create(new_content, metadata=existing.metadata, infer=False)
```

Expose via existing `PUT /memories/{id}` — callers don't have to know.

### Chunk reassembly

Memories with `metadata.chunk` + `metadata.total_chunks` get reassembled into a single virtual card in the UI. Backend stays unchanged; `frontend/src/lib/chunks.ts` groups by `source_file` + `chunk_group_id`. Edit on a reassembled card rewrites all chunks.

## Phase 2.0 — Hygiene + LLM adapters

### 1. Consolidation UI

- New tab or "Tools" panel item.
- List `source_file` groups with >1 memory.
- For each group: show count, oldest/newest timestamp, total chars, header preview.
- "Merge" action: dry-run preview (combined content + proposed metadata) → confirm → server does delete-and-add with `infer=False`. Chunk-aware.
- Replaces `scripts/consolidate_daily.py`. Script stays around as a CLI fallback.

### 2. Near-duplicate detection

- Backend endpoint `GET /memories/duplicates?threshold=0.9`.
- Runs Qdrant cosine similarity pass over existing embeddings (cheap — no new vectors computed).
- Returns pairs with similarity + both memory IDs.
- UI: sortable table, side-by-side diff, Keep-A / Keep-B / Merge / Dismiss actions.

### 3. Staleness flags

- Heuristic-based (no LLM required):
  - Age > 90 days, and
  - Content contains a "time-sensitive marker": words like `currently`, `today`, `this week`, a date older than the memory itself, or a Linear-style ticket ID.
- Surfaces as a badge on memory cards and a filterable "Stale" category chip.
- Dismiss action (sets `metadata.staleness_dismissed=true` so you can ignore known evergreens).

### 4. Metadata editor

- MemoryModal gains a Metadata tab.
- Shows full JSON, editable with a schema-hint UI for common keys (`source_file`, `type`, `project`).
- Save goes through the metadata-preserving path above.

### 5. LLM adapter layer

- Ships infrastructure only. No features using it yet.
- Settings UI: "AI features" section with provider dropdown + config fields + test button.

### Acceptance (2.0)

- [ ] Consolidation tab merges a multi-entry source_file in < 3 clicks, preview shown before destruction.
- [ ] Duplicates endpoint returns pairs in < 2s for 2k memories.
- [ ] Staleness flags appear without configuring an LLM.
- [ ] Metadata edits round-trip without losing any keys.
- [ ] `GET /features` reflects LLM state correctly when Ollama is on vs. off.

## Phase 2.1 — Ingest + better editor

### 6. Ingest pipeline

- New "Ingest" tab (hidden if no LLM configured).
- Input methods:
  - Paste text
  - Upload file (`.md`, `.txt`, `.pdf` via pdftotext, maybe `.docx`)
  - URL (fetches, strips chrome, passes body text)
- Pipeline:
  1. Chunk content (~6k chars per chunk, nomic-compatible)
  2. For each chunk, `LLMAdapter.extract(text, schema)` where schema is user-configurable (starter schema = project's `CLAUDE.md` / `STARTUP.md` pattern)
  3. Show extracted candidate memories in a review UI (Accept / Edit / Reject per item)
  4. Accepted items write to chosen adapter with metadata
- One-shot "Ingest a directory" button for filesystem adapter users.

### 7. Smart tagging (sponsorware-eligible)

- Add-memory flow optionally calls `LLMAdapter.tag()` to suggest metadata tags.
- Runs asynchronously — new memory appears immediately, tags populate after.
- User can accept / reject / edit per-tag.

### 8. Better markdown editor

- Replace textarea in `MemoryModal` with **CodeMirror 6** — syntax highlighting, frontmatter awareness, folding.
- Slash-command-free, no AI-completion — this is an editor, not a copilot.
- Preserves the keyboard-first feel of V1. Desktop shortcut `Cmd+S` saves.

### 9. Local → mem0 sync

- Button on any `fs:` card: "Sync to mem0".
- Writes content as a mem0 entry with `metadata.source_file` set to the file path and `metadata.synced_from` pointing to fs adapter id.
- If the file later changes, "Re-sync" prompts to overwrite the existing mem0 entry (metadata-preserving update).

### Acceptance (2.1)

- [ ] Ingest a markdown file into 3+ structured memories with configurable schema.
- [ ] CodeMirror editor handles 50k-char memory without lag.
- [ ] Sync button round-trips a local file to mem0 and back.

## Phase 2.2 — Analytics + export

### 10. Analytics tab

Keep it light — a few cards, not a BI dashboard:

- Total memories, by adapter, by `metadata.type`.
- Growth over time (monthly bars, last 12 months).
- Top source files by count.
- Staleness summary (stale count, dismissed count).
- **Coverage gaps** — stub for now (requires session data we don't have; design the empty state).

### 11. Export / backup

- `GET /export?adapter=mem0&format=json` — streams all memories as NDJSON.
- Also `format=markdown-zip` — one file per memory with frontmatter.
- CLI script `scripts/export.py` for cron usage.
- **Import:** paste JSON or upload NDJSON, adapter-aware, `infer=False`, duplicate-aware.

### Acceptance (2.2)

- [ ] Export 2k memories in < 5s.
- [ ] Re-import an export yields no net change (idempotent).
- [ ] Analytics loads in < 1s on 10k memories.

## Phase 2.3 — Multi-tenancy foundation

Don't build multi-user auth in V2. **Do** clean up every place that assumes single-user so the hosted version can land without a rewrite.

- Replace hardcoded `user_id` defaults (`"default"`, `"sage"`) with a config-level `DEFAULT_USER_ID` env var.
- Route `GET /memories` always scopes by `user_id` query param (already true; audit to ensure no leak paths).
- Workspace abstraction: new optional env `MEMVUE_WORKSPACE` (defaults to `default`). Stored alongside runtime config; UI shows current workspace in top bar (hidden if only one).
- Document the multi-user migration path in `docs/MULTI_TENANCY.md`.

### Acceptance (2.3)

- [ ] Running two backend instances with different `MEMVUE_WORKSPACE` values produces isolated adapter state.
- [ ] No code path references a hardcoded user id.

## Monetization hooks

Baked in, never in your face.

### In V2

- **Footer `♥ Support` link** — opens `https://buymeacoffee.com/<handle>` (or GitHub Sponsors). Configured via env `SUPPORT_URL`. Hidden if unset.
- **Settings → "About" section** — version info, sponsor link, "powered by memvue" with a link to the GitHub repo.
- **LLM provider affiliate slots** — when the user picks OpenRouter / Anthropic in Settings, show a "Don't have an account? Sign up here" link under the field. Configured via env so partners can be swapped without code changes.
- **Feature flag `sponsor_only`** — present in the code but never enforced in MIT core. Reserved for a future hosted build.

### Not in V2 (keep on the roadmap)

- Hosted memvue.app with Free / Pro / Team tiers.
- Commercial license alternative for companies.
- Prompt pack / taxonomy bundle marketplace.
- GitHub Sponsors "early access" release channel for AI features.

## Migration / breaking changes

- None for V1 users. `PUT /memories/{id}` gets a behavior change (metadata preserved), which is a bug fix, not a break.
- `runtime-config.json` schema gains `llm: {provider, config}` and `workspace: string`. Old configs work unchanged.

## Open questions

1. **Ingest schema config:** ship a starter schema set (journal, project log, research note), or let users write JSON Schema from scratch? Starter + custom is probably right.
2. **Duplicate threshold default:** 0.9 feels right for nomic-embed. Need to test.
3. **Should consolidation be automatic on ingest?** e.g. if I ingest a file that already has memories with that `source_file`, auto-merge? Lean no — opt-in surprise-free.
4. **Sponsor gating path:** MIT core exposes all features; a future hosted build uses `sponsor_only` flags. Alternative: sponsorware as a *release delay* (early-access branch for sponsors, main branch ungates after 90 days). Second option preserves OSS purity.

## Decision log

- **React, not vanilla** — confirmed in V1, keep.
- **LLM pluggable, not mandatory** — any provider via adapter, graceful degradation if unconfigured.
- **No open-core split** — sponsorware via release delay instead. Core features stay MIT.
- **Multi-user deferred** — V2 lays foundation, hosted version builds on it.
- **Buy Me a Coffee, not ads** — memory data is intimate. No ad surfaces, ever.
